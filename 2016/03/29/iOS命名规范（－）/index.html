iOS命名规范（－）

译自：[https://developer.apple.com/library/mac/documentation/Cocoa/Conceptual/CodingGuidelines/Articles/NamingBasics.html#//apple_ref/doc/uid/20001281-BBCHBFAH][1]

##基本原则：

  • 一个好的命名应该简明扼要。简洁的命名和单词的长度并不冲突,即使单词很长，也要尽量拼写完整
    | 命名 | |
    | ——– | —–: |
    | insertObject:atIndex: | 好的命名 |
    | insert:at: | 不好的命名,insert和at的含义太含糊 |

  • 尽量不要缩写，即使单词很长
    | 命名 | |
    | ——– | —–: |
    | destinationSelection | 好的命名 |
    | destSel | 含义不够清楚 |

  • 然而一些计算机术语的单词缩写，我们可以直接使用
    | 缩写 | 意义 |
    | ——– | —–: |
    | alloc | Allocate |
    | alt | Alternate |
    | app | application |
    | calc | Calculate |
    | dealloc | Deallocate |
    | msg | Message |
  • API方法的名字需要语义明确，不然会被别人错误的理解成其他意思，尤其是在不同的国家和文化里。
    | 代码 | 描述 |
    | ——– | —–: |
    | displayName | 不够明确，它即可被理解成表示用户名的文本又可被理解成界面里显示用户名的文本控件 |
    ##Consistency
    相同作用的方法的命名应该保持一致性；
    | 代码 | 描述 |
    | ——– | —–: |
    | - (NSInteger)tag | 在NSView, NSCell, NSControl中皆有声明 |
    | - (void)setStringValue:(NSString *) | 在很多Cocoa的类中都有这个方法 |

##不要自我解释
| 代码 | 描述 |
| ——– | —–: |
| NSString | 好的命名 |
| NSStringObject | 自我解释，NSString已经可以表示一个字符串对象，再加上object有些画蛇添足 |

但一些宏，按位操作的常量是个例外。
| 代码 | 描述 |
| ——– | —–: |
| NSUnderlineByWordMask | Okay |
| NSTableViewColumnDidMoveNotification | Okay |

##Prefix(前缀)
prefix在命名中是非常重要的一部分。它可以区分软件中不同的功能块，比如实现了一个功能模块的类库经常打包成一个framework。使用前缀可以避免第三方开发的类和苹果自己的类发生冲突。
前缀有固定的格式，一般由两三个大写的字母组成。比如Foundation使用”NS”作为前缀，AFNetworking使用”AF”作为前缀。
前缀用在命名类，协议，函数（c），常量和typedef结构体。而方法和变量无需添加前缀。

##书写约定
当一个方法由多个单词组成时，不要使用逗号等符号作为单词之间的分隔符。应使用骆驼命名法，即首个单词首字母小些，其他的单词首字母大些，例：fileExistsAtPath:isDirectory:
对于函数（c）和常量的名字，使用和类相同的前缀，并利用类中第一个单词，例：NSRunAlertPanel，NSCellDisabled

##类和协议的命名
类的名字中需包含一个用来说明这个类的作用的名字。比如：UIButton,UILabel,NSString.
协议应该根据它如何管理自己的行为来去命名：

  • 协议中的方法作用和具体的类无关。这样的协议命名应该和类名有明显的区别，避免让使用者误认为它是一个类。
    | 代码 | 描述 |
    | ——– | —–: |
    | NSUnderlineByWordMask | Okay |
  • 一些协议里有一些不相干的方法。这些协议一般是和一个类强相关的，这个类就是协议的实现者。这时，我们会把协议的名字设成和类的一致。比如NSObject协议。

##头文件

  • 声明一个单独的类或者协议。如果这个类或者协议并不是一个功能组的一部分，则将类或协议的声明放到一个单独的文件中，文件的名字即为该类或协议的名字。
    | 头文件 | 声明 |
    | ——– | —–: |
    | NSLocale.h | NSLocale类 |
  • 当声明一组功能相关的类，协议或者拓展时，将这些声明到一个文件当中，文件的名字为承载主要功能的类，协议，拓展的名字。
    | 头文件 | 声明 |
    | ——– | —–: |
    | NSString.h | 声明了NSString和NSMutableString类 |
  • 包涵framework的header files。每一个framework都应该包涵一个头文件，名字和framework的名字一样，里面包含了该framework里所有的头文件。
    | 头文件 | framework |
    | ——– | —–: |
    | Foundation.h | Foundation.framework |
  • 给别的frameowork的类添加api。如果你给别的framework的拓展类继续添加方法，那在给这个拓展命名时，需要在最原始的那个类的后面添加Additions。比如：NSBundleAdditions.h
